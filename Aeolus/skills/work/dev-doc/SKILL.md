# AI 生成前端开发评审文档

## 适用场景

- 用户要求根据模板生成前端开发评审文档
- 用户提供了 JIRA 任务链接、产品文档、Figma 设计链接等信息，要求写开发文档
- 用户提到"开发文档""评审文档""开发设计文档""前端评审"

## 模板来源

- 模板页面：[2026 前端开发文档模板](https://kms.fineres.com/pages/viewpage.action?pageId=1408374856)
- 模板 pageId：`1408374856`
- 如果用户指定了其他模板，按用户指定处理

## 默认规则

### 默认父页面

- 默认父页面：`数据中心任务`，pageId = `1407064611`
- 除非用户明确指定其他父页面，否则新文档挂到这个页面下

### 标题规则

- 如果用户给了标题，直接使用
- 如果用户只给了 JIRA 任务号和名称，标题格式为：`{JIRA任务号} {任务名称}`
- 示例：`BI-194155 禁用提示场景完善`

## 执行流程

### 1. 收集用户输入

从用户消息中提取以下信息（缺少的项在文档中留空）：

| 模块 | 说明 |
|------|------|
| 任务链接 | JIRA 任务链接或任务号 |
| 产品文档 | 产品设计文档链接 |
| 交互文档 | Figma 交互设计链接 |
| 视觉文档 | Figma 视觉设计链接或同交互文档 |
| 代码实现 | 代码仓库、分支、关键实现等 |

### 2. 获取模板内容

调用 `confluence_get_page` 获取模板页面内容：

```json
{
  "pageId": "1407064611"
}
```

### 3. 获取 Figma 设计信息（如有）

如果用户提供了 Figma 链接：

1. 使用 Figma MCP 的 `get_figma_data` 获取设计结构和节点信息
2. 使用 `download_figma_images` 获取设计截图
3. 将设计信息用于填充"功能拆解""组件及其层次管理"等章节

### 4. 生成文档内容

基于模板结构，按以下章节逐一填充：

#### 一. 基本信息
- 填入链接表格（任务链接、产品文档、交互文档、视觉文档、任务类型）
- **任务背景**：根据用户提供的任务信息和产品文档，概述现状和问题
- **设计目标**：简明扼要描述任务要达成的设计目标是什么样的。例如：重新实现仪表板跳转，解决潜在问题并不引入新问题。
- **竞品分析**：如有相关信息

#### 二. 详细设计
- **详细设计图**：根据 Figma 设计和产品文档，梳理功能点。如为重构任务则根据情况填充
- **引用/依赖资源**：如有相关信息
- **功能设计**：提供至少 2 个方案对比（如果有代码实现信息），在【功能详细设计】中对选定方案进行详细描述
- **关键代码**：代码结构
- **升级兼容设计**：如有相关信息

#### 三. 其它

根据模板选填测试研发需考虑的风险点

#### 四. 开发计划
- 开发耗时预估表格
- 转测时间

#### 五. 测试范围
**测试建议**
- 改动涉及范围，新老功能改动范围、架构改动范围	
- 自测用例、数据、配置等
- 重点需要测试的范围
- 期望测试额外覆盖的范围
- 性能测试建议，性能测试点
- 发散测试建议

### 5. 更新文档内容

如果是对存在的文档内容做更新，将更新内容简要概括到”改动记录“
例：
| 版本 | 改动内容 | 改动原因 | 改动日期 | 改动作者 |
|------|------|------|------|------|
v1 |	webui-bi 打包配置从 webpack 迁移到 rspack，新增 rspack/ 目录及相关配置文件 |	提升构建速度，降低开发等待时间 |	2026-03-19 |	Dailer |

### 6. 创建 KMS 页面

使用 `confluence_create_page` 创建文档：

```json
{
  "title": "文档标题",
  "content": "<生成的 Storage Format HTML>",
  "parentId": "1381752194"
}
```

### 7. 返回结果

创建成功后告知用户：
- 文档标题
- 文档链接
- 哪些章节已经自动填充，哪些需要手动补充

## 内容格式要求

- 使用 Confluence Storage Format HTML
- 链接表格使用 `<table>` 标签，保持与模板一致的列结构
- 测试建议表格保留 checkbox 任务项格式
- 方案对比使用四列表格：方案 / 实现原理 / 优点 / 缺点
- 如果信息不足，对应章节保留模板占位提示文字，不要编造内容

## 约束

- 不要编造用户未提供的技术细节
- 信息不足的章节保留模板提示，标注"待补充"
- 更新已有文档前必须先读取最新内容
- Figma 链接可选，没有也能生成文档框架
- 代码实现信息可选，有则填充方案设计和代码设计章节


## 示例

### 示例 1：用户提供完整信息

用户输入：

```text
帮我生成开发评审文档，信息如下：
任务链接：BI-194155 - 禁用提示场景完善
交互文档和视觉文档：https://www.figma.com/design/zKEyNrOLN1ufKkWeySN3sa/%E4%B8%89%E6%9C%88%E4%BD%93%E9%AA%8C%E4%BB%BB%E5%8A%A1%E5%90%88%E9%9B%86?node-id=147-16109&p=f&t=sWgG1GcKOy304H0I-0
```

处理方式：

- 获取模板内容
- 从 Figma 获取设计信息
- 生成标题：`BI-194155 禁用提示场景完善`
- 按模板结构填充各章节
- 创建到父页面 `1407064611` 下

### 示例 2：用户只提供任务号

用户输入：

```text
帮我创建 REPORT-12345 的前端开发文档
```

处理方式：

- 获取模板内容
- 生成标题：`BI-194155 前端开发评审文档`
- 按模板结构生成文档框架，各章节标注"待补充"
- 创建到父页面 `1407064611` 下
